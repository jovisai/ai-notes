# Build a Chat App That Uses Local Models for Easy Questions and Remote Models for Hard Ones

Every LLM call costs something — money, latency, or privacy. Send everything to GPT-4o and you burn budget on "what's the weather?" and "summarize this email." Run everything locally and you get frustrated when the model can't reason through a multi-step problem.

The fix is a **local-first routing gateway**: a thin layer that sends simple queries to a local model and escalates hard ones to a remote frontier model. Done right, 70–80% of queries never leave your machine. This article walks through the architecture, the routing strategies, and a concrete build using open-source tools.

---

## The Core Idea

A gateway sits between your chat UI and every model. It classifies each incoming query, then routes it to the cheapest tier that can handle it.

```
User query → Gateway → [Classifier]
                          ├── Easy → Local model (Ollama, llama.cpp)
                          └── Hard → Remote model (GPT-4o, Claude, Gemini)
```

The user sees one response. They don't know — or care — which model produced it.

---

## Three Routing Strategies

### 1. Confidence Cascading (Most Popular)

The local model generates a response plus a confidence score. If confidence is below a threshold, the gateway discards the local response and sends the query to the remote model.

**Pros:** Simple to implement. The local model never blocks the user — if it's unsure, the remote model takes over.

**Cons:** You always pay the local inference cost, even for hard queries. Latency doubles on escalated requests (local attempt + remote retry).

**When to use it:** You have a fast local model (3B–8B parameters) and can afford the double-pass on hard queries.

### 2. Classifier-First

A small classifier — or the local model itself with a system prompt — tags the query as "simple" or "complex" before any generation happens. The gateway routes based on that tag.

**Pros:** No wasted local inference on hard queries. Lower latency on the remote path.

**Cons:** You need to train or tune the classifier. Misclassifications send hard queries to the local model (bad answers) or easy ones to the remote model (wasted cost).

**When to use it:** You have labeled query data from an existing chat app and can train a lightweight classifier.

### 3. Two-Pass with Verification

The local model answers first. A separate verifier model (or the same model with a verification prompt) checks the answer quality. If it fails, fall back to remote.

**Pros:** Higher quality than raw confidence thresholds. The verifier catches hallucinations and incomplete answers.

**Cons:** Most expensive path — three model calls for hard queries (local + verify + remote).

**When to use it:** Quality is more important than cost or latency. Good for customer-facing support bots.

---

## What Makes "Easy" vs "Hard"?

This is the hardest part of the system. Here are signals people use:

| Signal | How it works | Caveat |
|---|---|---|
| **Response length** | Short answers → easy, long → hard | Brittle. A simple question can have a long answer. |
| **Token-level uncertainty** | Low probability tokens → low confidence | Requires access to logprobs. Not all local models expose them cleanly. |
| **Hedging language** | "I'm not sure", "It depends", "Based on my training" | Simple regex check on the local model's output. |
| **Query complexity score** | Classifier trained on query text | Needs labeled data. Best long-term approach. |
| **Task type** | Classification → local. Code generation → remote. | Rule-based. Works well if your use cases are well-defined. |

The most robust production approach combines **task-type routing** (rule-based, zero cost) with **confidence cascading** (fallback when the local model is uncertain).

## Building It: Stack and Code

### The Stack

| Component | Tool |
|---|---|
| **Local model server** | Ollama (easiest), llama.cpp, or vLLM |
| **Unified gateway** | LiteLLM proxy (recommended) or custom FastAPI |
| **Remote models** | OpenAI, Anthropic, Google, or any OpenAI-compatible API |
| **Routing logic** | LiteLLM config + a small Python router, or ModelCascade |

### Option A: LiteLLM Proxy (Simplest)

LiteLLM runs as a proxy server that speaks the OpenAI API. You configure your models in a YAML file, and any OpenAI-compatible client can use it.

**config.yaml:**

```yaml
model_list:
  - model_name: fast-local
    litellm_params:
      model: ollama/llama3.2:3b
      api_base: http://localhost:11434
      timeout: 30

  - model_name: smart-cloud
    litellm_params:
      model: openai/gpt-4o-mini
      api_key: os.environ/OPENAI_API_KEY

  - model_name: heavy-cloud
    litellm_params:
      model: anthropic/claude-sonnet-4-6
      api_key: os.environ/ANTHROPIC_API_KEY

router_settings:
  routing_strategy: usage-based  # or latency-based, cost-based
  allowed_fails: 3
  cooldown: 60
```

Start the proxy:

```bash
litellm --config config.yaml --port 4000
```

Now any client can call `fast-local` or `smart-cloud` through the same endpoint. Your app code decides which model name to use.

### Option B: Custom Python Gateway with Confidence Routing

For more control, build a small FastAPI gateway:

```python
from fastapi import FastAPI
from litellm import completion
import re

app = FastAPI()

LOCAL_MODEL = "ollama/llama3.2:3b"
REMOTE_MODEL = "gpt-4o-mini"

HEDGE_PATTERNS = [
    r"i'?m not sure",
    r"i don'?t know",
    r"it depends",
    r"based on my training",
    r"i cannot",
    r"i'?m (just )?an ai",
]

def is_low_confidence(text: str) -> bool:
    text_lower = text.lower()
    return any(re.search(p, text_lower) for p in HEDGE_PATTERNS)

@app.post("/chat")
async def chat(query: dict):
    user_message = query["message"]

    # Try local first
    local_response = completion(
        model=LOCAL_MODEL,
        messages=[{"role": "user", "content": user_message}],
        api_base="http://localhost:11434"
    )
    local_text = local_response.choices[0].message.content

    # If local seems confident, return it
    if not is_low_confidence(local_text) and len(local_text) > 20:
        return {"response": local_text, "model": "local"}

    # Otherwise escalate to remote
    remote_response = completion(
        model=REMOTE_MODEL,
        messages=[{"role": "user", "content": user_message}]
    )
    return {
        "response": remote_response.choices[0].message.content,
        "model": "remote"
    }
```

### Option C: ModelCascade (Purpose-Built Router)

There's an open-source project called **ModelCascade** that does exactly this. It's a Python library with a YAML config:

```yaml
# mc.yaml
providers:
  local:
    type: ollama
    model: llama3.2:3b
    cost_per_1k: 0.0
  fast:
    type: anthropic
    api_key: ${ANTHROPIC_API_KEY}
    model: claude-haiku-4-5
    cost_per_1k: 0.001
  capable:
    type: anthropic
    api_key: ${ANTHROPIC_API_KEY}
    model: claude-sonnet-4-6
    cost_per_1k: 0.005

routing:
  cost_ceiling: 0.01
  cascade_on_failure: true
```

Usage:

```python
from modelcascade import CascadeRouter

router = CascadeRouter.from_config("mc.yaml")
result = await router.complete(prompt)
# → routed LOCAL · $0.000 · 47ms
```

The library handles classification, confidence checking, and escalation automatically. Reports from production show 74% of requests handled locally at zero cost.

## Production Considerations

**Fallback in both directions.** Cloud APIs go down. Local GPUs saturate. Your gateway should detect failures and route around them — cloud downtime triggers local fallback, local saturation triggers cloud overflow.

**Cost tracking.** LiteLLM and ModelCascade both log per-request cost. Without this, you can't measure whether your routing is actually saving money.

**Latency budgets.** If your local model takes 2 seconds and your remote model takes 1 second (network included), routing to local is a net loss. Profile both paths before setting thresholds.

**Privacy.** Local-first means sensitive data never leaves your network for the majority of queries. Document which queries go remote so your compliance team can sign off.

**Routing decision caching.** If the same query pattern (e.g., "summarize this email") always escalates, cache the routing decision and skip the local attempt next time.

## Start Small

1. Install Ollama and pull a 3B model (`ollama pull llama3.2:3b`)
2. Set up LiteLLM proxy with one local and one remote model
3. Write a simple router that checks response length or hedging language
4. Log every routing decision and review misclassifications
5. Iterate on your routing logic

You don't need a classifier or a complex cascade on day one. A length-based heuristic will catch 60% of the savings. Add sophistication as you learn where your system breaks.
